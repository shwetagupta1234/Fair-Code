Fair Code is a research and engineering project that exposes bias in real-world AI systems and demonstrates concrete mitigation strategies. Both projects follow the same structure.

Train a biased model. Measure the fairness gap. Engineer a fair model. Measure again. No theory. Just data, code, and results.

The bias in these systems is documented, measurable, and fixable. Removing a protected attribute isn't enough. Proxy variables carry the same signal through. Both must go.

EXPLAINER — 01

WHAT IS A PROXY VARIABLE?

The sneaky reason AI stays biased even after you remove race from the data.

One-Sentence Definition

A proxy variable is a data feature that correlates with a protected attribute — like race, gender, or class — even though it doesn't mention it directly. So removing the protected attribute doesn't remove the bias it encodes.

Why This Matters

Most people assume that removing race from an AI model makes it race-neutral. This is wrong. Machine learning models don't care what a feature is called. They care what it predicts. If a feature like custody_status or zip_code is correlated with race in the training data, the model will learn and use that correlation — whether or not the word "race" appears anywhere in the dataset. This is called proxy discrimination, and it's one of the hardest problems in algorithmic fairness.

Common Proxy Variables
Feature Encodes Why
zip_code Race Historical redlining segregated neighbourhoods. Zip codes still reflect this.
custody_status Race Over-policing of Black communities leads to disproportionate pretrial detention.
name Gender / Ethnicity Names encode gender and ethnicity with high accuracy.
college_attended Socioeconomic class Elite colleges remain heavily stratified by class and race.
prior_arrests Race Arrest records reflect over-policing, not actual crime rates.
employment_tenure Age A 24-year-old structurally cannot have 10 years of work history.

How to Detect Proxy Variables
detect_proxy.py
from scipy.stats import chi2_contingency

def check_proxy(df, feature, protected_attr):
  """Returns p-value. If < 0.05, likely a proxy."""
  contingency = pd.crosstab(df[feature], df[protected_attr])
  chi2, p, dof, _ = chi2_contingency(contingency)
  return { 'p_value': round(p, 4), 'is_proxy': p < 0.05 }

# Run on every feature before training
result = check_proxy(df, 'CustodyStatus', 'race')
# → { 'p_value': 0.0001, 'is_proxy': True }
The Bigger Picture

Proxy variables exist because our social world is stratified. Zip codes encode race because of redlining. Custody records encode race because of over-policing. Employment tenure encodes age because time is linear. Data doesn't exist in a vacuum — it reflects the society that generated it. A model trained on that data will learn those reflections, unless you actively audit and intervene.

Algorithmic auditing is not optional. It is a prerequisite for deployment.

EXPLAINER — 02

WHAT IS SAMPLING BIAS?

The reason your AI works great in the lab and fails on the people who need it most.

One-Sentence Definition

Sampling bias occurs when the data used to train an AI system doesn't accurately represent the population it will be deployed on — so the model learns patterns that don't generalise, performing well for over-represented groups and failing for under-represented ones.

Why This Matters

Most people assume that if you have a large dataset, the model will be fair. This is wrong. Size doesn't equal representativeness. A dataset of 10 million records is still biased if 9.8 million of them come from the same demographic. The model doesn't know what it hasn't seen — it optimises for the distribution it was trained on. When that distribution doesn't match the real world, the people left out pay the price. This is how facial recognition fails on darker skin tones, clinical tools miss diagnoses in women, and NLP models perform worse on dialects outside Standard American English.

Common Forms of Sampling Bias
Form What It Means Real Example
Underrepresentation A group appears far less in training data than in the real world Facial recognition trained mostly on light-skinned faces
Selection Bias Data collected from a non-random subset of the population Clinical AI trained on records from affluent ZIP codes only
Historical Bias Past data reflects past inequities; the model learns the inequity Hiring models trained on 10 years of male-dominated engineering hires
Survivorship Bias Only "successful" cases make it into the dataset; failures are invisible Loan models trained only on approved applicants, never denied ones
Measurement Bias The same reality is measured differently for different groups Over-policing creates more arrest records for Black communities — not higher crime
Real-World Proof: MIT Gender Shades (2018)

Joy Buolamwini & Timnit Gebru tested three commercial facial recognition systems on a dataset balanced across gender and skin tone. Every system performed dramatically worse on darker-skinned women — not because the algorithms were programmed to fail them, but because training datasets were overwhelmingly composed of lighter-skinned faces. The fix was not a new algorithm. It was a more representative dataset.

Group Best System Error Rate Worst System Error Rate
Lighter-skinned males ~0.8% ~3.4%
Lighter-skinned females ~1.7% ~7.1%
Darker-skinned males ~10.8% ~16.0%
Darker-skinned females ~20.8% ~34.7%
A 34-percentage-point gap — from the same model, on the same task, because of who was and wasn't in the training data.

How to Detect Sampling Bias
audit_representation.py
def audit_representation(df, group_col, expected_proportions):
  """Flag groups as OVER/UNDERREPRESENTED vs real-world."""
  proportions = df[group_col].value_counts(normalize=True)
  audit = pd.DataFrame({'dataset_proportion': proportions})
  audit['expected'] = pd.Series(expected_proportions)
  audit['ratio'] = (audit['dataset_proportion'] / audit['expected']).round(4)
  audit['flag'] = audit['ratio'].apply(
    lambda r: 'OVERREPRESENTED' if r > 1.5
              else 'UNDERREPRESENTED' if r < 0.67
              else 'OK'
  )
  return audit

# Run before every training job
real_world = {'men': 0.51, 'women': 0.49}
result = audit_representation(df, 'gender', real_world)
# UNDERREPRESENTED → oversample, reweight, or restrict deployment
The Bigger Picture

Sampling bias is not a technical accident. It reflects who had access to the systems that generated the data in the first place. Facial recognition datasets skew lighter-skinned because the researchers who built them moved in predominantly lighter-skinned professional environments. Clinical AI skews wealthy because hospitals in affluent ZIP codes have better electronic health record infrastructure. NLP models skew toward Standard American English because that's what most of the training corpus is written in. The data you have is a record of the world as it was — including which people were visible, documented, and deemed worth measuring. A model trained on that data will reproduce those decisions at scale.

You cannot fix a representativeness problem with a better algorithm. You fix it with better data — or you disclose the limitation and restrict deployment accordingly.
